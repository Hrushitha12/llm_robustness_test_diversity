Robustness testing background:
Robustness testing evaluates system behavior in the presence of erroneous input
conditions. The goal is to stimulate internal errors by applying systematically
mutated input values to the parameters of each service call. The robustness of
the system is then classified according to the failure modes observed.

The following mutation rules define the categories of invalid inputs that should
be systematically applied to each parameter based on its data type:

String parameter mutations:
  - Replace by null value
  - Replace by empty string
  - Replace by a predefined string (a fixed known string unrelated to the parameter)
  - Replace by string with nonprintable characters
  - Add nonprintable characters to the string
  - Replace by alphanumeric string (where a specific format is expected)
  - Add characters to overflow the maximum expected size

Number parameter mutations:
  - Replace by null value
  - Replace by empty value (omit the parameter value)
  - Replace by -1
  - Replace by 1
  - Replace by 0
  - Add one to a valid value
  - Subtract one from a valid value
  - Replace by the maximum number valid for the type (e.g., Integer.MAX_VALUE)
  - Replace by the minimum number valid for the type (e.g., Integer.MIN_VALUE)
  - Replace by the maximum valid value plus one (overflow)
  - Replace by the minimum valid value minus one (underflow)

In addition to mutating parameter values, robustness tests should also consider
structural mutations to the request itself:
  - Omitting a required parameter key entirely from the request
  - Sending a request with no body at all
  - Sending a parameter with only its key and no value

Failure classification follows the CRASH scale:
  - Catastrophic: server crashes or reboots
  - Restart: service hangs and must be forcefully terminated
  - Abort: abnormal termination, such as an unexpected HTTP 500 response
  - Silent: no error indicated on an operation that cannot be performed
  - Hindering: incorrect error code or delayed response
In this study, we detect Abort-class failures (HTTP status >= 500).

Examples of robustness tests applying these mutation rules to TeaStore:

@Test
void test_R1_category_nullId() throws Exception {
    // Number mutation: replace by null value (omit the parameter key entirely)
    HttpResponse<String> r = get("/tools.descartes.teastore.webui/category");
    assertNoServerError(r);
}

@Test
void test_R1_category_negativeOneId() throws Exception {
    // Number mutation: replace by -1
    HttpResponse<String> r = get("/tools.descartes.teastore.webui/category?id=-1");
    assertNoServerError(r);
}

@Test
void test_R1_product_emptyId() throws Exception {
    // Number mutation: replace by empty value
    HttpResponse<String> r = get("/tools.descartes.teastore.webui/product?id=");
    assertNoServerError(r);
}

@Test
void test_R1_login_nonprintableUsername() throws Exception {
    // String mutation: replace by string with nonprintable characters
    HttpResponse<String> r = post("/tools.descartes.teastore.webui/loginAction",
        "username=\u0000\u0001\u0002&password=test");
    assertNoServerError(r);
}

@Test
void test_R1_cartAdd_keyAbsentProductId() throws Exception {
    // Structural mutation: omit required parameter key entirely
    HttpResponse<String> r = post("/tools.descartes.teastore.webui/cartAction",
        "addToCart=");
    assertNoServerError(r);
}

@Test
void test_R1_cartRemove_emptyProductId() throws Exception {
    // Number mutation: replace by empty value
    HttpResponse<String> r = post("/tools.descartes.teastore.webui/cartAction",
        "removeProduct=&productid=");
    assertNoServerError(r);
}

@Test
void test_R1_login_overflowUsername() throws Exception {
    // String mutation: add characters to overflow max size
    String longStr = "A".repeat(5000);
    HttpResponse<String> r = post("/tools.descartes.teastore.webui/loginAction",
        "username=" + longStr + "&password=test");
    assertNoServerError(r);
}

Notice: each test applies exactly one mutation rule from the taxonomy above to one
parameter of one endpoint. The comment above each test identifies which mutation
rule is being applied. Key-absent (omitting the parameter key entirely) and
value-empty (key present but value is empty string) are structurally different
mutations and must be tested separately.
